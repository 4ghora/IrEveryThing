## 1. Alert Name

**Device - ICMP Address Scan**

---

## 2. Alert Description (Detection Logic Explanation)

### 2.1 What triggered the alert

This alert is triggered when a device sends ICMP Echo Requests (Type 8) to multiple sequential or distributed IP addresses within a defined time window, deviating from its normal communication baseline.

**Detection Logic May Include:**

* **Behavioral anomaly detection** (Darktrace AI Model):

  * Device historically communicates with <10 IPs per hour.
  * Suddenly sends ICMP Echo Requests to 100+ internal IPs in 2 minutes.
* **Flow-based detection**:

  * High number of unique destination IPs.
  * Low packet size (typical ICMP echo payload ~32–64 bytes).
  * Consistent inter-packet timing.
* **Rate-based threshold logic**:

  * > X unique IPs in Y seconds.
* **Protocol-based anomaly**:

  * ICMP used internally where device normally uses only TCP/HTTPS.
* **Reconnaissance pattern detection** (MITRE ATT&CK T1018 – Remote System Discovery, T1046 – Network Service Discovery).

Example trigger:

* Host 10.10.15.34 sends ICMP Echo Requests to 10.10.15.1–10.10.15.254 in 60 seconds.

This is packet/flow-level detection, not payload-based.

---

### 2.2 Which security tool generated it

Generated by **Darktrace** (NDR platform using unsupervised machine learning).

Darktrace:

* Builds device behavioral profiles.
* Detects deviations in communication patterns.
* Does not rely only on signatures — uses AI-based baselining.

---

### 2.3 Why this alert is important from a network security perspective

ICMP address scanning is commonly used during:

* Initial compromise reconnaissance
* Lateral movement preparation
* Internal asset mapping
* Ransomware pre-encryption staging

Network reconnaissance is typically the **first step before exploitation**.

If malicious, this behavior indicates:

* Internal attacker presence
* Malware performing automated discovery
* Compromised endpoint probing network

Unchecked scanning often precedes:

* SMB exploitation
* RDP brute force
* Credential dumping operations
* C2 staging

---

### 2.4 Define Severity & Escalation Conditions

**Default Severity:** Medium

**Escalate to High if:**

* Source is a server (especially Domain Controller)
* Source is in PCI/CDE or production VLAN
* Scanning followed by:

  * SMB (445) connections
  * RDP attempts
  * LDAP queries
  * Kerberos failures
* Endpoint has EDR alerts
* Known compromised user logged in
* Scan targets critical subnets (DCs, backup servers, hypervisors)

**Escalate to Critical if:**

* Combined with privilege escalation
* Combined with C2 traffic
* Combined with data staging or exfiltration
* Ransomware-related indicators present

## 3. Knowledge Required Before Investigation

### 3.1 Concepts Analyst Must Understand

This section is critical for proper investigation.

---

### A. ICMP Protocol (Internet Control Message Protocol)

ICMP operates at Layer 3 (Network Layer). It is not a transport protocol like TCP or UDP.

Common ICMP Types:

* Type 8 – Echo Request (ping request)
* Type 0 – Echo Reply
* Type 3 – Destination Unreachable
* Type 11 – Time Exceeded

ICMP Echo Request structure:

* Type (8)
* Code (0)
* Identifier
* Sequence number
* Payload (optional data)

Example:

```
Host A → ICMP Echo Request → 10.10.10.20
Host B → ICMP Echo Reply → Host A
```

ICMP is:

* Stateless
* Connectionless
* Often allowed internally

Because it lacks authentication and logging depth, it is frequently abused for:

* Reconnaissance
* C2 tunneling
* Data exfiltration (ICMP tunneling tools)

---

### B. Network Scanning Concepts

**Address Scan vs Port Scan**

Address Scan:

* Identifies live hosts.
* Uses ICMP or ARP.
* Low noise compared to TCP scans.

Port Scan:

* Identifies open services.
* Uses TCP SYN, TCP Connect, or UDP.

Typical Attack Chain:

1. ICMP sweep → find live hosts.
2. TCP SYN scan → find open services.
3. Exploit service.

---

### C. Internal vs External ICMP

Internal ICMP scanning is more concerning because:

* Indicates internal foothold.
* Perimeter controls bypassed.
* Likely post-compromise behavior.

External ICMP scanning:

* Could be internet scanning.
* Often blocked by firewall.

---

### D. Legitimate ICMP Use Cases

Not all ICMP scanning is malicious.

Examples:

* Network monitoring tools
* SolarWinds-type health checks
* IT troubleshooting
* Asset discovery systems
* Backup verification

Analyst must differentiate:

* Scheduled scanning server vs user laptop
* Predictable vs sudden anomalous behavior

---

### E. MITRE ATT&CK Mapping

* T1018 – Remote System Discovery
* T1046 – Network Service Discovery
* T1083 – File and Directory Discovery (post scan)
* TA0007 – Discovery
* TA0008 – Lateral Movement (if followed by exploitation)

---

### F. Darktrace Detection Model Understanding

Darktrace creates:

* Peer group models
* Device role models
* Protocol usage baselines
* Communication entropy analysis

Deviation = Alert.

Example:

* Printer starts scanning subnet → High anomaly.
* Vulnerability scanner scanning nightly → Low anomaly if baseline exists.

## 4. Attacker Perspective

### 4.1 Why attackers use ICMP scanning

* Fast identification of live hosts
* Minimal logging
* Blends with legitimate traffic
* Often not blocked internally

---

### 4.2 What They Try to Achieve

* Map subnet topology
* Identify:

  * Domain Controllers
  * File servers
  * Backup systems
  * Admin workstations
* Prepare for lateral movement
* Identify pivot targets

---

### 4.3 Real-World Examples

* **WannaCry**

  * Performed internal scanning before propagation.
* **NotPetya**

  * Aggressively scanned internal networks.
* Cobalt Strike beacons

  * Perform host discovery before lateral exploitation.

---

### 4.4 Potential Business Impact

* Lateral compromise of entire domain
* Ransomware outbreak
* Credential harvesting
* Operational downtime
* Data breach

---

## 5. Pre-Investigation Checklist

### 5.1 Confirm Source/Destination Details

* Source IP
* Source MAC
* Hostname
* Logged-in user
* DHCP mapping
* Switch port
* Destination subnet
* Number of unique IPs targeted

---

### 5.2 Check Network Segment Criticality

* Is source in:

  * User VLAN?
  * Server VLAN?
  * DC VLAN?
* Is traffic:

  * Internal → Internal?
  * Internal → DMZ?
  * Internal → Cloud?

---

### 5.3 Verify Alert Context

* Are there other alerts from same host?
* Is there unusual SMB?
* Any RDP attempts?
* Any Kerberos anomalies?
* Any DNS anomalies?

---

## 6. Investigation Steps

### 6.1 Questions Analyst Should Ask

1. Is this device expected to perform network scanning?
2. Is the scanning pattern sequential or random?
3. How many unique IPs were targeted?
4. What happened after scanning?
5. Is there authentication activity after scan?
6. Is EDR reporting suspicious process?
7. Is this user privileged?

---

### 6.2 Answers to Questions

1. If device is vulnerability scanner → likely benign.
2. Sequential scanning suggests automation.
3. > 50 IPs in short window = suspicious.
4. SMB/LDAP/RDP following scan = high risk.
5. Kerberos failures indicate lateral attempt.
6. PowerShell process + ICMP = strong indicator.
7. Admin account increases severity.

---

### 6.3 Major Investigations

* Full packet capture review (if available)
* Check for:

  * TCP SYN scans after ICMP
  * SMB connections
  * NTLM authentication attempts
* Correlate with:

  * Windows Event ID 4624 / 4625
  * Sysmon Event ID 3
* Check EDR for:

  * nmap
  * PowerShell scripts
  * Living-off-the-land binaries

---

### 6.4 Minor Investigations

* Check change management (was IT troubleshooting?)
* Check vulnerability scanning schedule
* Validate monitoring tools
* Check recent patch deployment scripts

---

## 7. Evidence to Collect

* PCAP of ICMP traffic
* NetFlow logs
* Endpoint process tree
* User logon logs
* DHCP logs
* Switch port mapping
* Firewall logs
* EDR telemetry
* Darktrace model deviation report

---

## 8. Indicators of True Positive

* Sequential scanning of entire subnet
* Non-IT workstation performing scan
* Followed by SMB/RDP attempts
* PowerShell or nmap execution
* Concurrent EDR alert
* Privileged account logged in
* Scan targets critical infrastructure

---

## 9. Indicators of False Positive

* Approved vulnerability scanner
* Monitoring server performing health checks
* Network team troubleshooting
* Small number of IPs (<10)
* Known asset discovery tool

---

## 10. Incident Response Actions (If True Positive)

### 10.1 Containment

* Isolate endpoint from network
* Disable user account (if suspicious)
* Block at switch/firewall
* Snapshot system (forensics)

---

### 10.2 Eradication

* Remove malware
* Patch exploited vulnerabilities
* Reset credentials
* Remove persistence mechanisms

---

### 10.3 Recovery

* Reimage if necessary
* Monitor for recurrence
* Re-enable network access gradually
* Continuous monitoring for 72+ hours

---

## 11. Mitigation & Prevention

* Network segmentation
* Block ICMP between VLANs (if not required)
* Deploy EDR everywhere
* Enable Windows Firewall restrictions
* Monitor ICMP rate anomalies
* Implement NAC
* Least privilege enforcement
* Regular vulnerability scanning (authorized only)

---

## 12. Actions an IR Should Never Do (In Context of Alert)

* Do NOT immediately block ICMP globally (may break monitoring)
* Do NOT ignore internal scanning alerts
* Do NOT shut down critical servers without validation
* Do NOT alert suspected attacker user prematurely
* Do NOT rely solely on Darktrace without cross-validation
* Do NOT wipe system before collecting forensic evidence
