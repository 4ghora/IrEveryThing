## 1. Alert Name

**Device – Increase in new RPC services**

---

## 2. Alert Description (Detection Logic Explanation)

This alert indicates that a device has started exposing or interacting with an unusually high number of new RPC (Remote Procedure Call) services compared to its established behavioral baseline.

### 2.1 What Triggered the Alert

**Detection Type:** Behavioral anomaly detection (unsupervised machine learning) by Darktrace.

Darktrace models “normal” service exposure patterns for each device. The alert is triggered when:

* A host begins registering new RPC interfaces.
* A device suddenly initiates connections to multiple new RPC endpoints.
* RPC service UUIDs (interface identifiers) are observed that were not previously associated with this device.
* Abnormal volume of RPC endpoint mapper (TCP 135) lookups.
* Increased DCE/RPC binds and alter-context operations.

**Packet/Flow/Payload Level Examples:**

* **Packet-level:** Spike in TCP 135 sessions with DCE/RPC bind requests.
* **Flow-level:** Increase in internal-to-internal connections over high ephemeral ports (49152–65535).
* **Payload-level:** New UUIDs observed in DCE/RPC bind packets.
* SMB + RPC combinations (e.g., named pipe `\PIPE\svcctl`, `\PIPE\lsarpc`, `\PIPE\samr`).

---

### 2.2 Which Security Tool Generated It

Generated by **Darktrace NDR platform** using behavioral anomaly detection and protocol analysis (DCE/RPC dissection).

---

### 2.3 Why This Alert is Important

RPC is heavily used in Windows environments for:

* Active Directory replication
* Remote service creation
* WMI execution
* Scheduled task creation
* DCOM operations

A sudden increase in new RPC services may indicate:

* Lateral movement
* Privilege escalation
* DCSync-like behavior
* Remote service abuse
* Malware deployment

RPC is commonly abused in post-exploitation phases (MITRE ATT&CK:

* T1021.002 – SMB/Windows Admin Shares
* T1021.003 – Distributed Component Object Model
* T1047 – WMI
* T1003.006 – DCSync)

---

### 2.4 Severity Definition & Escalation Conditions

**Initial Severity:** Medium

**Escalate to High/Critical if:**

* Source host is workstation initiating RPC to Domain Controllers.
* RPC communication targets `lsarpc`, `samr`, `drsuapi`.
* Service creation via `svcctl` observed.
* Lateral spread across multiple hosts.
* Correlation with credential theft alerts.
* Activity targeting Tier-0 assets (Domain Controllers, AD CS, Backup servers).

## 3. Knowledge Required Before Investigation

### 3.1 Core Network and Protocol Concepts (Detailed)

An analyst must deeply understand:

---

### A. DCE/RPC (Distributed Computing Environment RPC)

RPC allows a client to execute code on a remote system.

**Key Components:**

* **Endpoint Mapper (EPM):** TCP 135
* Maps RPC interface UUIDs to dynamic ports.
* Client connects to 135 → receives dynamic port → reconnects to service.

**Flow Example:**

1. Client → Server: TCP 135 (RPC bind)
2. Server responds with dynamic port (e.g., 49672)
3. Client reconnects to 49672

Abnormal Behavior:

* Multiple new UUIDs requested.
* High number of new dynamic ports used.
* Excessive endpoint enumeration.

---

### B. RPC over SMB (Named Pipes)

RPC can operate over SMB using named pipes:

* `\PIPE\svcctl` – Service Control Manager (remote service creation)
* `\PIPE\lsarpc` – LSA policy access
* `\PIPE\samr` – SAM database queries
* `\PIPE\netlogon` – Authentication
* `\PIPE\drsuapi` – AD replication (DCSync abuse)

**Why Important:**
Attackers use named pipes to execute lateral movement tools.

---

### C. Windows Service Control Manager (SCM)

Remote service creation is common in:

* PsExec-style attacks
* Cobalt Strike beacons
* Ransomware deployment

Traffic Indicators:

* SMB session established
* RPC bind to `svcctl`
* Service create request
* Binary dropped in ADMIN$ share

---

### D. Active Directory Replication (DCSync)

Abuse of `drsuapi` RPC interface allows attackers to extract password hashes.

Network indicators:

* RPC from non-DC to DC
* Calls to `DRSGetNCChanges`

This is highly abnormal unless from a DC.

---

### E. Ephemeral Port Behavior

Windows uses 49152–65535 for dynamic RPC services.

Sudden spikes in ephemeral ports between two hosts can indicate:

* Lateral movement
* WMI execution
* DCOM exploitation

---

### F. MITRE ATT&CK Mapping (Network View)

| Technique               | Network Indicator           |
| ----------------------- | --------------------------- |
| Lateral Movement        | RPC binds to multiple hosts |
| Credential Dumping      | drsuapi to DC               |
| Remote Service Creation | svcctl usage                |
| C2 via DCOM             | Repeated DCOM RPC sessions  |

## 4. Attacker Perspective

### 4.1 Why Attackers Use RPC

RPC is:

* Native Windows protocol (blends in)
* Rarely blocked internally
* Required for legitimate admin tasks
* Difficult to distinguish without behavioral baseline

---

### 4.2 Attacker Objectives

* Move laterally
* Dump credentials
* Execute remote payloads
* Maintain persistence via services
* Abuse AD replication

---

### 4.3 Real-World Examples

* **Emotet** used RPC for lateral spreading.
* **TrickBot** abused DCOM and RPC.
* **Ryuk** used RPC service creation.
* **Conti** leveraged Windows services via RPC.

---

### 4.4 Potential Business Impact

* Domain compromise
* Credential theft
* Ransomware spread
* Full Active Directory takeover
* Business outage

## 5. Pre-Investigation Checklist

### 5.1 Validate Network Entities

* Source IP/MAC
* Destination IP/MAC
* Hostname mapping
* User logged in
* DHCP lease validation

---

### 5.2 Network Segment Criticality

* Is traffic internal-to-internal?
* Is DC involved?
* Is it crossing VLAN boundaries?
* Is it within PCI/CDE or production subnet?

---

### 5.3 Correlation

* Check for:

  * SMB anomalies
  * Kerberos failures
  * Brute-force alerts
  * Endpoint AV alerts
  * EDR suspicious process execution

---

## 6. Investigation Steps

### 6.1 Questions Analyst Should Ask

1. Is the source host an admin workstation?
2. Is it normal for this host to perform remote management?
3. Which RPC interfaces were accessed?
4. Was remote service creation performed?
5. Did this activity propagate to multiple hosts?
6. Are Domain Controllers involved?
7. Is there suspicious authentication preceding RPC?

---

### 6.2 Answers to 6.1

1. If workstation → suspicious.
2. If server management jump host → possibly normal.
3. Check UUID mapping (drsuapi, svcctl, samr).
4. Look for service creation patterns.
5. Multi-host spread indicates worming behavior.
6. DC involvement → escalate immediately.
7. Failed logins followed by RPC → credential attack.

---

### 6.3 Major Investigations

* Extract RPC UUIDs from Darktrace logs.
* Validate if drsuapi was invoked.
* Inspect SMB session setup.
* Check for ADMIN$ file writes.
* Query SIEM:

  * Event ID 4624 (logon type 3)
  * Event ID 7045 (service installed)
  * Event ID 4672 (privileged logon)
* Analyze network flows for lateral movement pattern.

---

### 6.4 Minor Investigations

* Check patch level of source host.
* Verify change management for new software.
* Confirm if vulnerability scanning occurred.
* Validate backup software RPC behavior.

---

## 7. Evidence to Collect

* PCAP of RPC traffic
* UUID identifiers
* SMB logs
* Windows Event Logs (Security + System)
* NetFlow records
* EDR process tree
* Service creation logs
* Authentication logs
* DHCP logs

---

## 8. Indicators of True Positive

* Workstation initiating drsuapi to DC.
* Service installed remotely.
* Multiple hosts contacted sequentially.
* Correlation with credential dumping.
* ADMIN$ write followed by svcctl bind.
* Non-admin user performing RPC service changes.

---

## 9. Indicators of False Positive

* Patch deployment server
* SCCM activity
* Backup software using RPC
* Vulnerability scanner performing enumeration
* Domain controller replication traffic between DCs

---

## 10. Incident Response Actions (If True Positive)

### 10.1 Containment

* Isolate source host immediately.
* Block RPC traffic from compromised host.
* Disable suspected account.
* Segregate affected subnet if spreading.

---

### 10.2 Eradication

* Remove malicious services.
* Reset credentials.
* Rebuild compromised machines.
* Remove persistence mechanisms.

---

### 10.3 Recovery

* Restore from clean backups.
* Monitor for reoccurrence.
* Enable enhanced logging.
* Revalidate AD integrity.

---

## 11. Mitigation & Prevention

* Restrict RPC access via segmentation.
* Block unnecessary internal RPC.
* Implement PAWs (Privileged Access Workstations).
* Enable SMB signing.
* Monitor RPC UUID anomalies.
* Alert on non-DC drsuapi usage.
* Implement Zero Trust internal segmentation.
* Harden service creation permissions.
* Enable advanced Windows auditing.

---

## 12. Actions an IR Should Never Do (In Context of Alert)

* Do NOT reboot host before evidence collection.
* Do NOT block Domain Controller replication blindly.
* Do NOT disable network-wide RPC without impact assessment.
* Do NOT assume admin activity is benign.
* Do NOT ignore ephemeral port anomalies.
