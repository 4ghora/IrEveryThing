## 1. Alert Name

**Device: New or Uncommon WMI Activity**

---

## 2. Alert Description (Detection Logic Explanation)

This alert indicates that **Darktrace NDR** observed Windows Management Instrumentation (WMI)-related network traffic that deviates from the learned baseline behavior of the device. The activity may represent lateral movement, remote execution, or remote enumeration using WMI over DCOM/RPC.

WMI network activity typically involves:

* **DCOM over RPC**
* TCP **135 (RPC Endpoint Mapper)**
* High ephemeral ports (49152–65535 on modern Windows)
* SMB (TCP 445) in some execution chains

Unusual patterns include:

* A workstation initiating WMI to multiple hosts
* A user workstation initiating WMI to domain controllers
* First-time WMI use from a device
* Off-hours WMI execution
* Abnormal frequency or spread of RPC/DCOM connections

---

### 2.1 What Triggered the Alert

This alert is typically triggered via **behavioral anomaly detection** at the **flow level**, sometimes enriched with protocol awareness:

**Examples of trigger logic:**

* A device that has *never previously used WMI remotely* initiates RPC connections to multiple internal hosts.
* Spike in TCP 135 connections followed by dynamic high port connections.
* Sudden east-west RPC traffic to servers not previously contacted.
* New device-to-device WMI communication pair never seen before.
* Abnormal connection fan-out pattern (possible automated execution).

**Packet/Flow-level indicators:**

* TCP SYN to port 135 → RPC bind request
* Follow-up connection to high dynamic port
* Short-lived sessions to many hosts (possible scanning/lateral movement)
* Repeated RPC bind attempts

Darktrace uses:

* Machine learning behavioral baselines
* Device “pattern of life” modeling
* Peer-group comparison (similar endpoints)

### 2.3 Why This Alert Is Important

WMI is heavily abused in post-exploitation phases.

From a network perspective:

* WMI enables **remote command execution**
* Often used for **lateral movement**
* Frequently leveraged in **ransomware propagation**
* Used for stealthy persistence and remote enumeration

MITRE ATT&CK relevance:

* T1047 – Windows Management Instrumentation
* T1021.003 – Remote Services: DCOM
* T1071 – Application Layer Protocol
* T1105 – Ingress Tool Transfer

Uncommon WMI activity may indicate:

* Credential compromise
* Privilege escalation
* Lateral movement inside internal network

---

### 2.4 Define Severity

**Default Severity: Medium–High**

Escalate to **High or Critical** if:

* Source host is a **workstation initiating WMI to multiple servers**
* Target includes **Domain Controllers**
* Off-hours activity
* Simultaneous alerts (SMB brute force, Kerberos anomalies)
* Large volume of RPC fan-out
* Device previously flagged for suspicious behavior
* WMI activity followed by data exfiltration
* Source host is in sensitive subnet (PCI/CDE, Server Farm)

# 3. Knowledge Required Before Investigation

## 3.1 Concepts Analyst Must Understand

This section is critical.

### A. Windows Management Instrumentation (WMI)

WMI is a Windows administrative framework allowing:

* Remote command execution
* System enumeration
* Process creation
* Service manipulation

Remote WMI uses:

* DCOM (Distributed Component Object Model)
* RPC

Network behavior:

1. TCP 135 → RPC Endpoint Mapper
2. Server responds with dynamic high port
3. Client connects to dynamic port
4. RPC communication begins

---

### B. RPC & DCOM (OSI Model Context)

* Operates at **Layer 7 (Application)**
* Uses **Layer 4 TCP**
* Often seen as:

  * TCP 135
  * TCP high dynamic port

RPC is stateful:

* TCP 3-way handshake required
* Binds to UUID interface
* Opens secondary port for session

Analyst must understand:

* Handshake success vs failed
* Connection duration
* Byte counts
* Flow direction

---

### C. East-West vs North-South Traffic

WMI lateral movement = **East-West traffic** (internal to internal).

Red flags:

* Workstation → multiple servers
* Workstation → Domain Controller
* Server → peer servers (unexpected role)

Normal baseline:

* SCCM server → endpoints
* Monitoring server → infrastructure

---

### D. Baseline Behavior

Darktrace relies on:

* Historical device behavior
* Peer group modeling

Example:
If an HR workstation never used RPC/DCOM before, first-time WMI is suspicious.

If SCCM server uses WMI daily → expected.

---

### E. Flow Analysis vs DPI

Flow data shows:

* Source IP
* Destination IP
* Ports
* Bytes
* Duration

DPI/Packet Capture reveals:

* RPC bind request
* UUID interface
* DCOM negotiation

---

### F. Beaconing Patterns

Though WMI is usually lateral movement, attackers may:

* Execute remote payload
* Establish C2 afterward

Look for:

* Follow-up HTTPS beaconing
* Repeated jittered traffic

---

### G. JA3 / TLS Fingerprinting (If Follow-Up C2 Seen)

If WMI triggers payload execution and new TLS traffic appears:

* Compare JA3 hashes
* Inspect SNI field
* Analyze certificate anomalies

---

### H. Practical Network Example

Normal:
SCCM server (10.1.10.5) → multiple workstations via 135 + high ports during business hours.

Suspicious:
User-PC-23 → Domain Controller + SQL server + file server within 2 minutes.

* No malware needed initially
* Bypasses perimeter detection
* Hard to distinguish from admin traffic

## 4.2 What They Try to Achieve

* Deploy C2 agents

# 6. Investigation Steps

## 6.1 Questions Analyst Should Ask

1. Has this device used WMI remotely before?
2. How many unique destinations were contacted?
3. Was TCP handshake successful?
4. Were dynamic ports used?
5. How many bytes transferred?
6. Is the target sensitive (DC/DB)?
7. What is time correlation?
8. Was there follow-up C2?
9. Is destination host compromised too?

---

## 6.2 Methodology to Answer

Using NDR / Zeek:

* Check conn.log for:

  * tcp.port == 135
  * Duration
  * Orig_bytes / Resp_bytes

Wireshark filter:

```
tcp.port == 135
```

Check dynamic RPC:

```
tcp.port >= 49152
```

In SIEM:

* Count distinct destination IPs
* Group by 5-minute window

Check authentication logs:

* 4624 (logon type 3)
* 4672 (privileged logon)

---

## 6.3 Major Investigations

### 1. Flow Pattern Analysis

* High fan-out?
* Sequential host access?
* Rapid connection attempts?

### 2. DPI / PCAP Analysis

* RPC bind request UUID
* Check DCOM object
* Identify WMI CLSID

### 3. Credential Abuse

* Correlate with Kerberos TGT issuance
* NTLM authentication spikes

### 4. Follow-Up Payload Execution

* Check SMB file transfer
* New service creation
* HTTP beaconing after WMI

---

## 6.4 Minor Investigations

* DNS queries before activity
* Firewall logs (allow/deny)
* Proxy logs
* Endpoint EDR telemetry
* Patch management activity validation

---

# 7. Evidence to Collect

* PCAP (full session)
* NetFlow/IPFIX
* Zeek conn.log + dce_rpc.log
* Firewall logs
* AD Security logs
* EDR process tree
* DHCP lease logs
* Switch port logs

---

# 8. Indicators of True Positive

* Workstation initiating WMI to multiple servers
* Off-hours activity
* RPC to Domain Controller
* Rapid fan-out pattern
* Successful authentication logs
* Follow-up SMB file transfer
* Subsequent external C2 beaconing
* Large lateral byte transfer

---

# 9. Indicators of False Positive

* SCCM server patch deployment
* Monitoring system checks
* Backup server activity
* IT admin performing maintenance
* Vulnerability scanner using WMI
* Newly deployed management software

---

# 10. Incident Response Actions (If True Positive)

## 10.1 Containment

* Network isolate source host (NAC/EDR)
* Block source IP via firewall ACL
* Disable compromised user account
* Null route suspicious internal traffic

---

## 10.2 Eradication

* Reset credentials
* Remove persistence
* Scan lateral targets
* Patch exploited vulnerabilities

---

## 10.3 Recovery

* Restore affected systems
* Validate integrity
* Monitor for re-infection
* Increase telemetry logging

---

# 11. Mitigation & Prevention

* Strict internal segmentation
* Limit RPC/DCOM access via firewall
* Disable unnecessary WMI remote access
* Implement least privilege
* Monitor RPC baseline deviations
* Deploy Zero Trust Network Access
* Enforce MFA for admin accounts
* Use EDR with lateral movement detection
* Block dynamic RPC from user VLANs to server VLANs

---

# 12. Actions an IR Should Never Do

* Do NOT blindly block TCP 135 without impact analysis
* Do NOT assume all WMI is malicious
* Do NOT alert the user before containment
* Do NOT reboot compromised host before collecting memory
* Do NOT tip attacker by immediate global firewall rule without monitoring
* Do NOT ignore east-west traffic anomalies
