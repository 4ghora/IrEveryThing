## 1. Alert Name

**SaaS – Compromise – Login from Rare Location Following Suspicious Login Attempts**

---

## 2. Alert Description (Detection Logic Explanation)

This alert indicates that a user account successfully authenticated to a SaaS application from a **rare or anomalous geographic location** shortly after multiple suspicious login attempts (e.g., failed attempts, brute-force behavior, impossible travel anomalies, or login attempts from flagged IPs).

---

### 2.1 What Triggered the Alert

This alert is typically triggered by **behavioral anomaly detection** combined with login pattern analysis.

Darktrace likely identified:

* Multiple failed login attempts (high frequency, abnormal velocity, or suspicious ASN/IP reputation).
* Followed by a successful login from:

  * A rare country or ASN for that user.
  * A new device fingerprint.
  * An IP associated with TOR/VPN/hosting provider.
  * Anomalous time-of-day deviation.

Detection layers:

* **Flow-level metadata** (HTTPS session establishment to SaaS provider).
* **Behavioral deviation** (user profile baseline vs current login).
* **GeoIP anomaly detection**.
* **Impossible travel logic** (e.g., login from India, then 20 minutes later from Eastern Europe).
* **Reputation intelligence** (threat IP database).

Example triggers:

* 12 failed login attempts from ASN in Russia.
* 1 successful login 3 minutes later from same ASN.
* User historically logs in only from India corporate IP.

### 2.3 Why This Alert Is Important from a Network Security Perspective

This alert suggests **potential account compromise**.

From network perspective:

* SaaS compromise can bypass traditional perimeter controls.
* Attackers may use valid credentials (T1078 – Valid Accounts).
* Compromised SaaS accounts enable:

  * Data exfiltration (T1567 – Exfiltration Over Web Services)
  * Business Email Compromise (BEC)
  * Lateral movement via OAuth apps
  * Persistence through token abuse

Unlike malware-based attacks, this is **identity-driven intrusion** — extremely dangerous and stealthy.

---

### 2.4 Severity & Escalation Conditions

**Default Severity: High**

Escalate to Critical if:

* Successful login from TOR/VPN hosting ASN.
* MFA bypass detected.
* OAuth token created.
* Admin role account affected.
* Large data download after login.
* Suspicious mailbox rules created.
* Login targeting finance or executive account.
* Concurrent anomalous network traffic (e.g., bulk HTTPS uploads).

## 3. Knowledge Required Before Investigation

### 3.1 Concepts Analyst Must Understand

This section is critical.

---

### A. HTTPS Authentication Traffic (TLS Handshake + Web Authentication)

Most SaaS login traffic occurs over:

* TCP 443
* TLS 1.2/1.3 encrypted sessions
* HTTP POST requests carrying authentication payload

Important concepts:

**TLS Handshake**

* ClientHello (cipher suites, SNI, TLS fingerprint)
* ServerHello
* Certificate exchange

Even though payload is encrypted, metadata is visible:

* SNI (Server Name Indication)
* JA3/JA3S fingerprint
* IP/ASN
* Session duration
* Bytes transferred

Example:
User device connects to:

* login.microsoftonline.com
* accounts.google.com
* salesforce.com

Analyst must understand:

* What normal authentication traffic looks like.
* Typical session size for login vs data transfer.

---

### B. GeoIP & ASN Profiling

GeoIP maps IP to country.
ASN identifies the network owner.

Examples:

* Residential ISP in India (normal).
* Hosting ASN in Netherlands (abnormal).
* TOR exit node ASN (high risk).

Analyst must understand:

* VPN/Cloud providers (DigitalOcean, OVH, Hetzner).
* Corporate VPN exit IP ranges.
* Proxy infrastructure behavior.

---

### C. Authentication Attack Patterns

1. Password spraying (low & slow across many users).
2. Credential stuffing (many attempts quickly).
3. Brute force.
4. MFA fatigue attack.
5. OAuth token abuse.

Network indicators:

* High frequency login attempts.
* Multiple usernames from same IP.
* Repeated HTTPS sessions with small payload size.

---

### D. Impossible Travel Detection

Example:

* Login from Mumbai at 10:00.
* Login from Germany at 10:15.

Based on IP distance & time difference.

---

### E. SaaS API Abuse & Data Exfiltration

After login, attacker may:

* Download mailbox via API.
* Download SharePoint/Drive files.
* Create forwarding rules.
* Upload malicious OAuth app.

Network signs:

* High-volume HTTPS download.
* Long-duration TLS sessions.
* Repeated GET requests.

---

### F. Identity Baseline Modeling

Darktrace builds:

* Typical login hours.
* Typical IP ranges.
* Typical device fingerprint.
* Typical browser TLS signature.

Deviation triggers anomaly.

## 4. Attacker Perspective

### 4.1 Why Attackers Use This Technique

Because:

* No malware needed.
* Bypasses EDR.
* Hard to detect if credentials are valid.
* SaaS is outside corporate firewall.

---

### 4.2 What They Try to Achieve

* Establish persistent access.
* Steal email data.
* Financial fraud (BEC).
* Data exfiltration.
* Pivot into internal systems via SSO.

MITRE mapping:

* T1078 – Valid Accounts
* T1110 – Brute Force
* T1567 – Exfiltration Over Web Services
* T1098 – Account Manipulation

---

### 4.3 Real-World Examples

* Microsoft 365 BEC campaigns.
* Google Workspace credential stuffing attacks.
* OAuth consent phishing.
* Attack campaigns by groups like **APT28** targeting cloud accounts.

---

### 4.4 Potential Business Impact

* Financial loss (wire fraud).
* Data leakage.
* Regulatory penalties.
* Reputational damage.
* Privilege escalation inside hybrid environments.

## 5. Pre-Investigation Checklist

### 5.1 Confirm Technical Identifiers

* Source IP
* Destination IP
* ASN
* Username
* Device ID
* User agent
* JA3 fingerprint
* MAC (if internal NAT exit known)

Correlate via:

* DHCP logs
* VPN logs
* AD logs
* SaaS audit logs

---

### 5.2 Network Segment & Directionality

Determine:

* Internal → SaaS
* External → SaaS
* VPN exit → SaaS

Check if:

* From corporate network
* From residential ISP
* From cloud provider

---

### 5.3 Concurrent Alerts

Check:

* Same IP hitting multiple users.
* Same user generating multiple alerts.
* Data exfil alerts after login.
* Endpoint alerts on user device.

---

## 6. Investigation Steps

---

### 6.1 Questions Analyst Should Ask

1. Is the successful login legitimate?
2. Does IP belong to user’s normal geography?
3. Were failed attempts brute-force behavior?
4. Was MFA used?
5. Was there data access after login?
6. Is IP malicious/reputation flagged?
7. Is login part of password spraying campaign?

---

### 6.2 Major Investigations

* SaaS audit logs (login, token creation).
* OAuth app registrations.
* Mailbox rule creation.
* Data download volume.
* Admin privilege changes.
* Check impossible travel events.
* Check for token refresh events (persistent session).

---

### 6.3 Minor Investigations

* DNS logs for suspicious domains.
* Proxy logs.
* CASB alerts.
* User endpoint EDR status.
* Check if IP is TOR exit node.

---

## 7. Evidence to Collect

* Full login event logs.
* Source IP & ASN info.
* User agent string.
* MFA logs.
* Session ID.
* Audit logs of:

  * File downloads.
  * Email rule changes.
  * Admin changes.
* Packet capture (if available).
* TLS metadata.

---

## 8. Indicators of True Positive

* IP from hosting provider.
* User denies login.
* MFA fatigue before success.
* Data download spike.
* Forwarding rule created.
* OAuth app added.
* Multiple failed attempts prior.
* Login outside business hours.

---

## 9. Indicators of False Positive

* User traveling internationally.
* Corporate VPN IP recently changed.
* ISP IP reassigned.
* Legitimate third-party SSO.
* User using new mobile device.
* New office location rollout.

---

## 10. Incident Response Actions (If True Positive)

### 10.1 Containment

* Immediately disable account.
* Revoke active sessions.
* Revoke refresh tokens.
* Force password reset.
* Block malicious IP at firewall.
* Remove malicious OAuth apps.

---

### 10.2 Eradication

* Reset credentials.
* Re-enroll MFA.
* Audit mailbox rules.
* Remove persistence artifacts.
* Investigate endpoint compromise.

---

### 10.3 Recovery

* Restore deleted emails/files.
* Monitor account closely for 30 days.
* Enable conditional access policies.
* Notify stakeholders.

---

## 11. Mitigation & Prevention

* Enforce MFA (phishing-resistant).
* Disable legacy authentication.
* Conditional Access (geo-restriction).
* Detect impossible travel.
* Monitor OAuth app consent.
* Rate-limit login attempts.
* Block TOR/proxy ASNs.
* Deploy CASB integration.
* Monitor login velocity anomalies.

---

## 12. Actions an IR Should Never Do

* Never notify attacker through compromised email.
* Never reset password without revoking tokens.
* Never ignore failed login patterns.
* Never assume VPN IP is legitimate without validation.
* Never close case without SaaS audit review.
* Never rely only on network logs without identity logs.
