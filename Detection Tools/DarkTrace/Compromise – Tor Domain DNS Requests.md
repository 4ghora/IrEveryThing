## 1. Alert Name

**Compromise – Tor Domain DNS Requests**

---

## 2. Alert Description (Detection Logic Explanation)

### 2.1 What Triggered the Alert

This alert is triggered when an internal host performs DNS queries for domains associated with the **Tor network**, particularly:

* Known **Tor directory authorities**
* Known **Tor relay nodes**
* Known **.onion-related resolver domains**
* Domains historically associated with Tor bootstrap infrastructure
* DNS queries to domains linked to Tor bridges

Detection may be based on:

* **Threat intelligence signature match** (Known Tor infrastructure domain list)
* **Behavioral anomaly detection** (Host previously never queried Tor-related domains)
* **DNS pattern recognition** (Repeated resolution attempts to Tor bootstrap servers)
* **Flow-level correlation** (DNS request followed by TLS session to known Tor entry node IP)

**Packet/Flow Level Examples:**

* DNS A record request:

  ```
  Query: torproject.org
  Query: directory.torproject.org
  Query: bridge.torproject.org
  ```
* Subsequent outbound TCP connection:

  ```
  TCP 443 -> Known Tor Guard Node IP
  TLS handshake with no SNI or uncommon certificate issuer
  ```

The detection may operate at:

* **Packet-level** (DNS payload inspection)
* **Flow-level** (DNS → TLS correlation)
* **Reputation feed correlation**

---

### 2.2 Detection Tool

Generated by: **Darktrace** (NDR platform)

Darktrace uses:

* Self-learning AI models
* Behavioral anomaly detection
* Threat intelligence correlation
* Device baseline deviation detection

Unlike signature-only IDS (e.g., Suricata), Darktrace may trigger this alert when:

* A host deviates from its historical DNS behavior
* A non-technical user endpoint suddenly queries Tor-related domains

---

### 2.3 Why This Alert Is Important

Tor usage inside enterprise networks is high-risk because:

* Tor is commonly used for:

  * Command & Control (MITRE ATT&CK T1071 – Application Layer Protocol)
  * Exfiltration over encrypted channels (T1041)
  * Defense evasion (T1090.003 – Multi-hop Proxy)
* Tor encrypts traffic and obscures destination
* Traditional firewalls cannot inspect Tor payloads
* Data exfiltration over Tor bypasses standard monitoring

Tor usage may indicate:

* Malware beaconing
* Insider attempting to bypass monitoring
* Data staging & exfiltration
* Ransomware pre-encryption staging

---

### 2.4 Severity & Escalation Conditions

**Default Severity: Medium–High**

Escalate to **High or Critical** if:

* DNS resolution is followed by:

  * Successful outbound connection to Tor guard node
  * Large outbound encrypted sessions
* Source host is:

  * Domain Controller
  * Database server
  * PCI/CDE asset
* Concurrent alerts exist:

  * Data exfiltration anomaly
  * Lateral movement
  * Privilege escalation
* Endpoint shows:

  * Suspicious process (tor.exe, powershell, unknown binary)
* Multiple hosts resolving Tor domains simultaneously

## 3. Knowledge Required Before Investigation

### 3.1 Core Concepts & Protocol Understanding

An analyst must deeply understand:

---

### A. DNS Protocol (Layer 7, UDP/TCP 53)

DNS translates domain names into IP addresses.

**Flow Example:**

1. Client → DNS Resolver:

   ```
   UDP 53 Query: directory.torproject.org
   ```
2. Resolver → Root/TLD → Authoritative Server
3. Resolver → Client:

   ```
   A Record: 116.x.x.x
   ```

Key DNS concepts:

* A/AAAA records
* NXDOMAIN responses
* Recursive resolution
* DNS logging (query name, response code, TTL)

Suspicious indicators:

* High frequency DNS queries
* Repeated failed lookups
* Rare domain entropy patterns
* Queries for anonymization infrastructure

---

### B. Tor Network Architecture

Tor = The Onion Router.

It operates via:

1. Client connects to:

   * Guard node
2. Traffic routed through:

   * Middle relay
3. Exit node:

   * Final destination

Tor bootstrap process:

* Client resolves directory authorities
* Downloads relay list
* Establishes encrypted circuit

Tor uses:

* TCP 9001
* TCP 9030
* TCP 443 (common)
* TLS-based encrypted communication

From network perspective:

* TLS traffic with:

  * No typical browser fingerprint
  * No standard SNI
  * Long-lived encrypted sessions

---

### C. NDR Detection Logic

Difference between:

| Detection Type   | Explanation                              |
| ---------------- | ---------------------------------------- |
| Signature-based  | Known Tor domain list match              |
| Behavioral       | Device deviation from baseline           |
| Threat Intel     | Known malicious Tor exit IP              |
| Flow correlation | DNS → Immediate TLS session to Tor guard |

Darktrace uses behavior modeling:

* “This host has never used anonymization networks before”

---

### D. MITRE ATT&CK Network Aspects

Relevant techniques:

* T1090.003 – Multi-hop Proxy (Tor)
* T1071.001 – Web Protocols
* T1041 – Exfiltration over C2 Channel
* T1573 – Encrypted Channel
* T1568 – Dynamic Resolution

---

### E. Enterprise Network Segmentation Awareness

Analyst must understand:

* Where the host resides (VLAN, subnet)
* Egress firewall rules
* Proxy enforcement
* DNS logging location
* SSL inspection status

Example:
If host bypasses corporate proxy and directly accesses Tor, this indicates policy evasion.

## 4. Attacker Perspective

### 4.1 Why Attackers Use Tor

Attackers use Tor to:

* Hide C2 infrastructure
* Prevent IP attribution
* Encrypt communication end-to-end
* Bypass IP reputation filtering
* Avoid geo-based blocking

---

### 4.2 Attacker Objectives

* Establish persistent C2
* Exfiltrate stolen data anonymously
* Stage ransomware communication
* Maintain stealth access
* Sell access on dark web

---

### 4.3 Real-World Examples

* **Ryuk ransomware** operators used Tor for C2
* **TrickBot** used Tor hidden services
* **Conti** data leak sites hosted on Tor
* Nation-state actors used Tor bridges to bypass restrictions

---

### 4.4 Potential Business Impact

* Sensitive data exfiltration
* Regulatory violations (PCI, HIPAA, GDPR)
* Ransomware deployment
* Intellectual property theft
* Long-term persistence

---

### 5.3 Alert Correlation

* Check for:

  * Data exfiltration alerts
  * Suspicious TLS anomalies
  * Lateral movement alerts
  * Multiple Tor-related events

---

## 6. Investigation Steps

### 6.1 Questions Analyst Should Ask

1. Did the DNS query succeed?
2. Was it followed by outbound connection?
3. How much data was transferred?
4. Is Tor client installed?
5. Is this user authorized to use Tor?
6. Is this behavior new for this device?

---

### 6.2 Answers to 6.1 (Investigation Logic)

* If DNS succeeded + TLS session established → Likely active Tor usage.
* If only DNS attempt with NXDOMAIN → Possible malware test beacon.
* If large outbound encrypted session → Possible exfiltration.
* If process = tor.exe → Confirmed Tor client.
* If powershell spawned connection → Malicious.

---

### 6.3 Major Investigations

* Pull full packet capture (if available)
* Check:

  * JA3 TLS fingerprint
  * SNI field
  * Certificate issuer
* Correlate with endpoint telemetry:

  * EDR process tree
  * Parent-child process chain
* Review:

  * Proxy logs
  * Firewall logs
  * NetFlow data
* Check for:

  * Data staging activity before event

---

### 6.4 Minor Investigations

* VirusTotal lookup of resolved IP
* WHOIS analysis
* Passive DNS history
* Check if exit node is known malicious
* Compare against enterprise Tor allowlist

---

## 7. Evidence to Collect

* EDR telemetry
* Process hash
* User login session logs
* Firewall logs
* TLS fingerprint data
* Data transfer size metrics

---

## 8. Indicators of True Positive

* Successful Tor guard node connection
* Tor TLS fingerprint match
* tor.exe running
* Large encrypted outbound session
* Host never used Tor before
* Concurrent suspicious activity

---

## 9. Indicators of False Positive

* Security research team using Tor legitimately
* Threat intelligence scanning lab
* User accessing torproject.org informational site only
* DNS prefetch by browser
* Security tool sandbox testing

---

## 10. Incident Response Actions (If True Positive)

### 10.1 Containment

* Isolate host from network
* Block Tor IP ranges at firewall
* Disable user account (if compromised)
* Revoke active sessions

---

### 10.2 Eradication

* Remove Tor client
* Remove persistence mechanisms
* Run full malware scan
* Reimage if required

---

### 10.3 Recovery

* Restore system from clean backup
* Reset credentials
* Monitor for re-infection
* Validate no lateral movement occurred

---

## 11. Mitigation & Prevention

* Block Tor directory authorities at DNS
* Implement DNS sinkhole
* Enforce proxy-only outbound access
* Enable SSL inspection where legally allowed
* Deploy egress filtering
* Enable DNS logging with retention
* Implement UEBA monitoring
* Monitor JA3 fingerprints
* Restrict admin privileges

---

## 12. Actions an IR Should Never Do (In Context of This Alert)

* Do NOT immediately block without confirming legitimacy
* Do NOT assume Tor = malicious without context
* Do NOT ignore DNS-only events
* Do NOT alert attacker by killing process prematurely before evidence capture
* Do NOT reimage before collecting forensic artifacts
* Do NOT rely solely on IP reputation (Tor nodes rotate)
