## 2. Alert Description (Detection Logic Explanation)

### 2.1 What Triggered the Alert

This alert is generated when **Antigena (Darktrace’s autonomous response engine)** automatically blocks or restricts suspicious traffic associated with a SaaS platform.

Detection is typically based on **behavioral anomaly detection**, not static signatures.

Trigger conditions may include:

* **Anomalous data upload volume** to a SaaS domain (flow-level anomaly).
* **Unusual SaaS access time** (e.g., 03:00 AM from a finance user).
* **First-time SaaS domain access** from a host.
* **Abnormal user-agent, TLS fingerprint (JA3), or HTTP headers**.
* **Geo-velocity anomaly** (login from India → 10 minutes later from EU).
* **OAuth abuse patterns**.
* **High-entropy uploads** (possible encrypted exfil payloads).
* **Rare API calls to SaaS service** (e.g., bulk export API).

Detection Levels:

* **Packet level**: abnormal TLS handshake, invalid cert chain, unusual SNI.
* **Flow level**: large outbound data spikes, new domain frequency.
* **Payload metadata level**: HTTP method anomalies (PUT/POST bulk upload).

Example:

* User normally accesses Microsoft 365 with 5MB daily upload.
* Suddenly uploads 2GB to a rarely used cloud storage SaaS.
* Darktrace detects deviation from behavioral baseline.
* Antigena blocks connection.

### 2.3 Why This Alert is Important (Network Security Perspective)

SaaS traffic is:

* Encrypted (TLS 1.2/1.3)
* Often trusted by firewalls
* Whitelisted in egress rules

Attackers exploit SaaS to:

* Exfiltrate sensitive data.
* Establish covert C2 over trusted domains.
* Abuse OAuth tokens.
* Stage payload delivery.

Blocking suspicious SaaS activity indicates:

* Potential **data exfiltration (MITRE T1041)**.
* Possible **Command & Control over web services (T1102)**.
* Account takeover.
* Insider threat.

SaaS-based attacks bypass perimeter IDS due to:

* Valid TLS certificates.
* Trusted domains (e.g., Dropbox, Google Drive).
* Legitimate ports (443).

---

### 2.4 Define Severity & Escalation Conditions

**Default Severity: Medium–High**

Escalate to High/Critical if:

* Large outbound data transfer (>500MB unusual).
* Communication with known malicious SaaS subdomain.
* Activity from Domain Admin / privileged account.
* Target is PCI/CDE or sensitive subnet.
* OAuth token abuse observed.
* Multiple hosts showing same behavior (campaign).

## 3. Knowledge Required Before Investigation

### 3.1 Network & Protocol Concepts Analyst Must Understand

This section is critical.

---

### A. SaaS Architecture (Network View)

SaaS operates over:

* HTTPS (TCP 443)
* TLS 1.2 / TLS 1.3
* API-based REST calls
* OAuth authentication

Traffic Flow:

```
Internal Host → DNS Query → Public SaaS IP → TLS Handshake → HTTPS Data Exchange
```

Key concepts:

* SNI (Server Name Indication)
* JA3 TLS fingerprint
* HTTP Methods (GET, POST, PUT, DELETE)
* API endpoints
* OAuth token exchange

---

### B. TLS Inspection Limitations

Most NDR tools:

* See metadata (IP, port, JA3, SNI)
* Not full decrypted payload (unless SSL inspection enabled)

Therefore detection relies on:

* Flow analytics (NetFlow/IPFIX)
* Behavioral baselining
* Frequency and size anomalies
* Certificate analysis

Example:
If host sends:

* 10MB daily to Microsoft 365
* Suddenly 1.5GB to unknown SaaS domain

Even without decryption, anomaly triggers.

---

### C. SaaS Abuse Techniques

1. Data Exfiltration via Cloud Storage
   Upload sensitive files to:

   * Google Drive
   * Dropbox
   * Mega
   * OneDrive

2. C2 over SaaS
   Malware uses:

   * Slack APIs
   * Discord webhooks
   * Google Sheets
   * GitHub repositories

3. OAuth Token Abuse
   Attacker registers malicious app.
   Victim grants permissions.
   Attacker accesses mailbox or files remotely.

---

### D. DNS & Domain Behavior

Analyst must understand:

* Domain reputation
* Newly registered domains
* Subdomain abuse
* CDN-backed SaaS IP resolution

SaaS often uses:

* CDN IP ranges
* Shared infrastructure (Cloudflare/Akamai)

So IP reputation alone is insufficient.

---

### E. Flow-Level Analysis (Critical)

Understand:

* Bytes sent vs received.
* Session duration.
* Beaconing intervals.
* Burst uploads.

Example of Exfil Pattern:

* Large outbound.
* Minimal inbound.
* Single long-lived session.

Example of C2:

* Small periodic POST every 60 sec.
* Consistent packet size.

## 6.2 Answers to Questions in 6.1 (Analytical Interpretation)

### 1. Is this SaaS domain legitimate?

Do **not** answer this with simple reputation lookup.

Instead determine:

* Is this domain officially approved by the organization?
* Has this host/user accessed this SaaS before?
* Is this SaaS part of shadow IT usage?

Even legitimate SaaS domains (e.g., Dropbox, Google Drive) can be abused for:

* Data exfiltration
* C2 over web services (MITRE T1102)
* Staging ransomware payloads

If it is:

* Newly observed SaaS for this user → suspicious.
* Rare SaaS domain across enterprise → higher risk.
* Newly registered SaaS subdomain → high risk.

---

### 2. Is the user authorized to use this SaaS?

Authorization must be evaluated from:

* Role-based access (Finance, DevOps, HR etc.)
* SaaS usage policy
* Conditional Access logs

If:

* Privileged account using non-approved SaaS → high severity.
* Service account making outbound SaaS calls → highly suspicious.
* Domain controller communicating to SaaS → critical escalation.

---

### 3. Is data transfer volume abnormal?

Compare:

* Bytes sent vs 30-day behavioral baseline.
* Upload-to-download ratio.
* Single burst vs multiple sessions.

Red flags:

* 5–10x baseline upload deviation.
* Outbound >> Inbound traffic ratio.
* Long-lived TLS session with steady outbound transfer.
* Large POST/PUT requests.

This strongly aligns with:

* Data Exfiltration (T1041)
* Archive Collected Data (T1560)

---

### 4. Is timing unusual?

Check:

* User login hours baseline.
* Is this after-hours?
* Weekend activity?
* Time-zone mismatch (geo anomaly)?

Example:
User typically active 9AM–6PM.
Now uploading 1.2GB at 02:30 AM → behavioral deviation.

If combined with:

* VPN from new geography → possible account compromise.

---

### 5. Is host showing other compromise indicators?

Correlate with:

* EDR alerts
* Suspicious parent process (powershell.exe, rundll32.exe)
* Credential dumping events
* DNS anomalies
* Lateral movement logs

If SaaS activity follows:

* SMB lateral movement
* Privilege escalation
* LSASS access

Then SaaS usage likely represents **final-stage exfiltration**.

---

### 6. Is TLS fingerprint (JA3) unusual?

Compare:

* JA3 hash to enterprise baseline.
* Known malware TLS fingerprints.
* Self-signed or rare cipher usage.

Red flags:

* Non-browser JA3 connecting to browser-expected SaaS.
* PowerShell TLS client impersonating Chrome.
* Curl/Wget user-agent to enterprise SaaS.

This may indicate:

* Scripted API abuse
* Malware C2 over HTTPS

---

### 7. Is there repeated beaconing behavior?

Look for:

* Consistent interval connections (e.g., every 60 sec).
* Small fixed-size HTTPS POST.
* Low data but regular timing.

That suggests:

* C2 channel (T1071.001 Web Protocols).
* Webhook-based communication.

If beaconing preceded bulk upload → staged exfiltration.

---

### 6.3 Major Investigations

* Analyze NetFlow:

  * Bytes sent vs received.
  * Session duration.
  * Frequency.
* DNS resolution history.
* EDR process initiating connection.
* Proxy logs.
* Authentication logs.
* Check for file compression before upload.
* Check for suspicious parent process (powershell, cmd, rundll32).

If upload spike observed → high suspicion of exfiltration.

---

### 6.4 Minor Investigations

* User travel history.
* VPN geo anomaly.
* Check SaaS app audit logs.
* Verify if company recently onboarded new SaaS.
* Confirm if backup software triggered upload.

## 8. Indicators of True Positive

* Large outbound transfer spike.
* High entropy uploads.
* Unapproved SaaS usage.
* Correlated malware alert.
* Suspicious PowerShell preceding upload.
* Data staged before upload.
* Repeated beaconing pattern.

---

## 9. Indicators of False Positive

* New legitimate SaaS onboarding.
* IT bulk data migration.
* Backup to cloud.
* User legitimately working after hours.
* Software update syncing files.
* Known enterprise SaaS expansion.

## 10. Incident Response Actions (If True Positive)

### 10.1 Containment

* Isolate host via EDR.
* Block domain/IP temporarily.
* Revoke OAuth tokens.
* Disable compromised account.
* Force password reset.

---

### 10.2 Eradication

* Remove malware.
* Delete unauthorized SaaS apps.
* Reimage host if needed.
* Patch vulnerabilities.
* Remove persistence mechanisms.

---

### 10.3 Recovery

* Restore from backup.
* Monitor SaaS logs.
* Enable stricter monitoring.
* Audit data exposure.
* Inform legal/compliance if data leaked.

---

## 11. Mitigation & Prevention

* SaaS allowlisting.
* CASB implementation.
* TLS inspection (if policy allows).
* DLP for SaaS.
* Egress filtering.
* OAuth app approval workflow.
* Conditional access policies.
* Zero Trust network access.
* Continuous NDR monitoring.
* User behavior analytics integration.

## 12. Actions an IR Should Never Do

* Do NOT immediately block SaaS globally.
* Do NOT reset account without collecting logs.
* Do NOT assume large upload = malicious.
* Do NOT ignore Antigena auto-block logs.
* Do NOT rely only on IP reputation.
* Do NOT close alert without baseline comparison.
* Do NOT isolate critical production server without coordination.
