## 1. Alert Name

**Antigina - Network - Insider threat - Antigina large data volume outbound block**

---

## 2. Alert Description (Detection Logic Explanation)

### 2.1 What triggered the alert

This alert is triggered by **behavioral anomaly detection** within Darktrace’s Antigena autonomous response module.

Specifically, it detects:

* **Unusual high-volume outbound data transfer**
* Deviation from historical traffic baseline for that host/user
* Abnormal destination (new external IP/domain)
* Data transfer outside normal business hours
* Protocol misuse or rare outbound service usage

Detection level may involve:

* **Flow-level analysis** (NetFlow/IPFIX-like telemetry: bytes, sessions, duration)
* **Packet metadata inspection** (TCP flags, session sizes, window scaling)
* **Behavioral modeling** (ML-based baseline comparison per device profile)

Example triggers:

* A finance workstation normally uploads <50 MB/day, but suddenly transfers 8 GB over HTTPS to an unknown ASN.
* Internal file server initiating outbound SFTP to a rare IP.
* DNS tunneling causing high outbound byte ratio via UDP/53.

This is not purely signature-based. It is anomaly-based with automated response (Antigena may have blocked or rate-limited traffic).

---

### 2.2 Which security tool generated it

Generated by: **Darktrace**

Specifically:

* Detection Engine: Enterprise Immune System (AI behavioral modeling)
* Response Module: Antigena (autonomous response engine)

---

### 2.3 Why this alert is important from a network security perspective

Large outbound data transfers are a primary indicator of:

* **Data Exfiltration (MITRE ATT&CK T1041, T1567)**
* Insider Threat activity
* Compromised host staging data for external transfer
* C2 channel with data retrieval
* Ransomware pre-encryption data theft

Outbound data volume is a critical network detection control because:

* Exfiltration must traverse the network
* Encryption does not hide flow metadata
* Volume and timing anomalies are detectable even when payload is encrypted (TLS)

---

### 2.4 Define Severity

**Default Severity: High**

Escalate to Critical if:

* Successful large-volume transfer confirmed (not just attempted)
* Destination IP belongs to:

  * Known malicious infrastructure
  * Bulletproof hosting
  * High-risk ASN
* Host is within:

  * PCI/CDE
  * Domain Controllers
  * Database servers
* Transfer occurred outside business hours
* Endpoint shows malware/C2 alerts
* Repeated anomaly after Antigena block

---

## 3. Knowledge Required Before Investigation

### 3.1 Concepts Analyst Must Understand

This alert revolves around **network traffic behavior**, not just malware signatures. The analyst must deeply understand:

---

### A. Flow vs Packet vs Payload Analysis

**Flow-based telemetry (NetFlow/IPFIX):**

* Source IP, Destination IP
* Ports
* Protocol
* Bytes in/out
* Duration
* TCP flags

Used to detect:

* High byte ratio
* Long-duration sessions
* Unusual external communication

Example:

```
Src: 10.10.15.23
Dst: 185.203.x.x
Protocol: TCP 443
Duration: 1h 12m
Bytes Out: 7.8GB
Bytes In: 30MB
```

High outbound-to-inbound ratio suggests exfiltration.

---

**Packet-level analysis:**

* TCP handshake patterns
* Retransmissions
* Window scaling
* Fragmentation
* TLS handshake metadata (SNI, JA3)

Used to detect:

* TLS fingerprint anomalies
* Non-browser TLS clients
* Data chunking behavior

---

**Payload-level inspection:**

* HTTP headers
* DNS queries
* File uploads
* Base64 encoding
* Multipart POST

Encrypted payloads limit visibility; thus, metadata analysis becomes critical.

---

### B. Outbound Data Exfiltration Techniques

Attackers may use:

1. HTTPS POST uploads
2. SFTP/FTP over non-standard ports
3. DNS tunneling (large TXT records)
4. Cloud storage abuse (Dropbox, Google Drive, Mega)
5. TOR relays
6. SMB over VPN tunnels
7. ICMP tunneling

Analyst must understand:

* Normal outbound traffic profile per subnet
* Proxy vs direct internet egress architecture
* SSL inspection deployment (if exists)

---

### C. Baseline Deviation Modeling

Darktrace builds device-specific behavioral baselines:

* Typical data volume per hour/day
* Typical peers
* Usual protocols
* Normal working hours
* Typical geolocation of destinations

Deviation examples:

* HR laptop suddenly communicating with foreign VPS
* Printer sending outbound TLS traffic
* Database server initiating DNS TXT floods

---

### D. Directionality & Network Segmentation

Must understand:

* Internal → External (most dangerous for exfil)
* Internal → Internal (possible staging/lateral movement)
* East-West vs North-South traffic

Segments:

* DMZ
* Server VLAN
* Endpoint VLAN
* Guest network
* Cloud VPC peering

Critical asset involvement increases severity.

---

### E. TCP Session Behavior in Exfiltration

Look for:

* Long-lived TCP sessions
* High outbound bytes
* Consistent packet sizes (data streaming)
* PSH flags dominant
* Stable window sizes

Attackers prefer:

* Stable TLS channel
* No retransmissions
* Mimicking browser behavior

---

## 4. Attacker Perspective

### 4.1 Why attackers use large outbound transfers

Because final objective is:

* Data theft
* Intellectual property extraction
* Credential dump exfiltration
* Database export removal

They must send data outside network boundary.

---

### 4.2 What they try to achieve

* Bypass DLP
* Blend into HTTPS traffic
* Use encrypted channels
* Evade signature IDS
* Exploit trusted cloud services

---

### 4.3 Real-world attack examples

* **Tesla insider data theft case (2018)**
* **Capital One breach (data exfil via cloud misconfiguration)**
* Ransomware groups like:

  * **LockBit**
  * **Clop**

Double extortion models require large outbound transfers before encryption.

---

### 4.4 Potential Business Impact

* Regulatory fines (GDPR/PCI)
* IP theft
* Legal liability
* Reputational damage
* Competitive disadvantage
* Stock impact

---

## 5. Pre-Investigation Checklist

### 5.1 Validate Identity

* Confirm Source IP
* Map to hostname (via DHCP logs)
* Map to user (via AD logs)
* Confirm MAC address consistency
* Check if device is corporate or BYOD

---

### 5.2 Network Segment & Direction

* Is traffic Internal → External?
* Is source in:

  * Finance VLAN?
  * Server farm?
  * Domain Controller subnet?
* Was traffic routed via proxy?

---

### 5.3 Alert Correlation

Check:

* Other Darktrace alerts for same host
* EDR alerts
* Proxy logs
* DLP alerts
* Firewall blocks
* VPN logs

Repeated anomaly = higher confidence.

---

## 6. Investigation Steps

### 6.1 Questions Analyst Should Ask

1. Is this data transfer legitimate?
2. Has this host historically transferred similar volume?
3. What protocol was used?
4. Is the destination known/trusted?
5. Was the transfer blocked or successful?
6. Is there evidence of staging before exfil?
7. Are there signs of C2 communication?
8. Is encryption used?
9. Is user behavior consistent with job role?

---

### 6.2 Answering the Questions

**Q: Is it legitimate?**
Check change management, backup schedules, cloud sync agents.

**Q: Historical pattern?**
Compare last 30–90 days of flow data.

**Q: Protocol used?**

* HTTPS?
* SFTP?
* DNS?
* Unknown TCP high port?

**Q: Destination?**

* Check WHOIS
* ASN reputation
* GeoIP
* Passive DNS

**Q: Blocked or successful?**

* Firewall logs
* TCP FIN vs RST behavior
* Session byte count

---

### 6.3 Major Investigations

* Full flow reconstruction
* PCAP review (if available)
* TLS fingerprint analysis (JA3)
* DNS log review for tunneling
* Proxy upload logs
* Endpoint EDR review
* Check for data compression prior to transfer
* Check PowerShell logs (if Windows)
* Check archive creation (7zip, WinRAR)

---

### 6.4 Minor Investigations

* Check user login times
* USB insertion logs
* Email attachments sent
* Cloud storage usage
* Check if device is recently onboarded

---

## 7. Evidence to Collect

* NetFlow/IPFIX logs
* Firewall session logs
* Proxy logs
* DNS logs
* DHCP logs
* PCAP (if possible)
* EDR telemetry
* Windows Event Logs (4688, 4624)
* File access logs
* Darktrace model breach summary

Preserve timestamps precisely.

---

## 8. Indicators of True Positive

* High outbound byte ratio
* Unknown external IP
* Rare ASN
* Long duration session
* Data compressed before transfer
* Unusual working hours
* Correlated EDR alert
* Repeated anomaly after block
* DNS tunneling patterns (high entropy subdomains)

---

## 9. Indicators of False Positive

* Legitimate backup upload
* Cloud migration
* Software update push
* Video conference data spike
* Developer pushing code to GitHub
* Approved large data sync

Must validate with asset owner.

---

## 10. Incident Response Actions (If True Positive)

### 10.1 Containment

* Isolate host (NAC or EDR isolation)
* Block destination IP/domain
* Disable user account (if insider suspected)
* Disable VPN session
* Block egress at firewall

---

### 10.2 Eradication

* Malware removal
* Credential reset
* Remove persistence mechanisms
* Reimage system if necessary
* Patch exploited vulnerabilities

---

### 10.3 Recovery

* Restore from clean backup
* Reintroduce host to network under monitoring
* Monitor for re-attempted exfil
* Increased logging on that subnet

---

## 11. Mitigation & Prevention

* Implement strict egress filtering
* SSL inspection (where legally allowed)
* DLP integration
* CASB for cloud control
* DNS logging + entropy detection
* Limit outbound SFTP
* Zero Trust segmentation
* User behavior analytics
* Monitor data staging indicators

---

## 12. Actions an IR Should Never Do (In Context of Alert)

* Do NOT immediately shut down the system without collecting volatile evidence.
* Do NOT delete logs.
* Do NOT alert suspected insider prematurely.
* Do NOT whitelist destination without investigation.
* Do NOT rely solely on volume — validate context.
* Do NOT assume encryption = safe traffic.
* Do NOT ignore internal staging before exfil.
