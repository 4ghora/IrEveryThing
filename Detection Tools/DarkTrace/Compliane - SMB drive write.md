## 1. Alert Name

**Compliance – SMB Drive Write**

---

## 2. Alert Description (Detection Logic Explanation)

### 2.1 What Triggered the Alert

This alert is typically triggered when a device performs a **write operation over SMB (Server Message Block)** to a network share, and the activity violates baseline behavior or compliance policy.

From a detection perspective, Darktrace may trigger this alert based on:

* **Behavioral anomaly detection** (self-learning AI model):

  * A host that has never written to network shares suddenly performs large SMB write operations.
  * A user account writes to multiple SMB shares across the environment in a short time.
  * A workstation writing to sensitive server shares outside normal business hours.
* **Flow-level analytics**:

  * Spike in SMB traffic (TCP 445) from a single source.
  * High byte count in SMB2 WRITE requests.
* **Protocol-level inspection (payload analysis)**:

  * SMB2 WRITE commands to administrative shares (`C$`, `ADMIN$`, `IPC$`).
  * File extensions indicating suspicious payloads (e.g., `.exe`, `.ps1`, `.dll`, `.bat`).
* **Compliance policy violation**:

  * Writes to restricted shares (e.g., finance, HR, PCI zone).
  * Writes originating from non-domain-joined systems.

Example triggers:

* A workstation sends repeated `SMB2 WRITE` requests to a file server storing backups.
* A compromised machine writes ransomware-encrypted files to shared drives.
* A service account begins writing executable payloads to multiple servers.

---

### 2.2 Which Security Tool Generated It

Alert generated by: **Darktrace**

Darktrace is an NDR platform using AI-based anomaly detection, focusing on behavioral deviation rather than purely signature-based detection.

---

### 2.3 Why This Alert Is Important from a Network Security Perspective

SMB write activity is directly associated with:

* **Lateral Movement (MITRE T1021.002 – SMB/Windows Admin Shares)**
* **Data Exfiltration (MITRE T1041 – Exfiltration over C2 Channel)**
* **Ransomware propagation**
* **Credential abuse and privilege escalation**

SMB is internal-facing and trusted by default in most enterprises. Abuse of SMB write operations often means:

* The attacker has **valid credentials**
* The attacker has **internal foothold**
* The attack is in **post-exploitation phase**

SMB-based threats are often invisible to perimeter firewalls because they occur east-west internally.

---

### 2.4 Define Severity & Escalation Conditions

**Default Severity:** Medium

**Escalate to High/Critical if:**

* Write operations target:

  * Domain Controllers
  * Backup servers
  * File servers with sensitive data
  * PCI/CDE subnet
* Large volume writes (>100MB unexpectedly)
* Writes followed by file rename patterns (ransomware behavior)
* Writes using administrative shares (`C$`, `ADMIN$`)
* SMB write followed by remote service creation or WMI execution
* Correlated alerts:

  * Credential dumping
  * Lateral movement alerts
  * Suspicious authentication spikes
  * C2 beaconing

## 3. Knowledge Required Before Investigation

### 3.1 Network & Protocol Concepts Analyst Must Understand

This section must be deeply understood before investigating.

---

### A. SMB Protocol (Server Message Block)

SMB is a stateful application-layer protocol running over:

* TCP 445 (direct hosting)
* TCP 139 (NetBIOS session service – legacy)

Modern environments use **SMB2 or SMB3**.

#### SMB Communication Flow:

1. TCP handshake (3-way)
2. SMB Negotiate Protocol Request
3. Session Setup (authentication via NTLM/Kerberos)
4. Tree Connect (connect to share)
5. Create/Open File
6. **WRITE Request**
7. Close

The WRITE operation contains:

* File ID
* Offset
* Data length
* Payload

From network perspective:

* High byte count in SMB2 WRITE → file upload
* Many small writes → ransomware encryption

---

### B. Windows Administrative Shares

Hidden shares:

* `C$`
* `ADMIN$`
* `IPC$`

These are used by administrators and attackers for:

* Remote file copy
* Service deployment
* PsExec-based execution

If a workstation writes to `\\Server\ADMIN$`, this is highly suspicious unless it's an IT admin device.

---

### C. Authentication Mechanisms

SMB uses:

* NTLM
* Kerberos

Analyst must correlate:

* Event ID 4624 (Logon Type 3 – Network Logon)
* NTLM authentication anomalies
* Service account abuse

If SMB write occurs without corresponding legitimate authentication pattern, it suggests:

* Pass-the-Hash
* Stolen Kerberos tickets
* Token impersonation

---

### D. Lateral Movement via SMB

Common attacker flow:

1. Initial compromise (phishing)
2. Credential dumping (Mimikatz)
3. SMB authentication to remote host
4. Write malicious payload
5. Create remote service
6. Execute payload

SMB write is often **step 4**.

---

### E. Ransomware Network Propagation Pattern

Network indicators:

* Multiple SMB sessions to many hosts
* Write operations to user shares
* File rename bursts
* High entropy files
* Spike in SMB traffic volume

---

### F. Normal vs Abnormal SMB Behavior

Normal:

* Workstations accessing file server shares
* Backup servers writing during scheduled window
* Software deployment tools (SCCM)

Abnormal:

* Peer-to-peer workstation SMB writes
* Finance server receiving writes from HR workstation
* Writes outside business hours
* Service account writing executable files

## 4. Attacker Perspective

### 4.1 Why Attackers Use This Network Technique

SMB is:

* Trusted internally
* Rarely blocked east-west
* Credential-based (blends with legitimate traffic)

Attackers use SMB writes to:

* Deploy malware
* Move laterally
* Stage data
* Encrypt shared drives

---

### 4.2 What They Try to Achieve

* Lateral movement
* Persistence
* Privilege escalation
* Data staging for exfiltration
* Ransomware deployment

---

### 4.3 Real-World Attack Examples

* **WannaCry** – Used SMB to propagate via EternalBlue exploit.
* **Ryuk** – Deployed via SMB after credential compromise.
* **NotPetya** – Used SMB and stolen credentials to spread laterally.

---

### 4.4 Potential Business Impact

* Enterprise-wide ransomware
* Data corruption
* Data exfiltration
* Domain-wide compromise
* Regulatory fines (PCI, HIPAA, GDPR)

## 5. Pre-Investigation Checklist

### 5.1 Validate Network Context

* Source IP
* Destination IP
* MAC addresses
* DHCP mapping
* AD user account involved
* Device hostname

---

### 5.2 Network Segment & Direction

Determine:

* Internal-to-Internal?
* Workstation-to-Server?
* Cross-segment?
* Into restricted VLAN?

---

### 5.3 Alert Correlation

Check:

* Multiple SMB alerts?
* Authentication anomalies?
* C2 detections?
* Endpoint alerts?

---

## 6. Investigation Steps

### 6.1 Questions Analyst Should Ask

1. Is this host normally writing to SMB shares?
2. Is the user account privileged?
3. What files were written?
4. What is the volume of data transferred?
5. Were administrative shares used?
6. Is this lateral movement?
7. Is this ransomware staging?

---

### 6.2 Answers to 6.1

* Check historical baseline in Darktrace.
* Check AD group membership.
* Inspect file extensions via packet capture.
* Compare byte count to baseline.
* Identify share path accessed.
* Correlate with remote service creation logs.

---

### 6.3 Major Investigations

* Extract SMB session metadata
* Analyze SMB2 WRITE frequency
* Correlate with Windows Event Logs (4624, 4672, 5145)
* Check for remote service creation (Event 7045)
* Check EDR telemetry

---

### 6.4 Minor Investigations

* Business justification?
* Maintenance window?
* Backup process running?
* Patch deployment in progress?

---

## 7. Evidence to Collect

* PCAP of SMB session
* NetFlow data
* Authentication logs
* File hash of written files
* Share access logs
* Endpoint process execution logs

---

## 8. Indicators of True Positive

* Write to `ADMIN$`
* Executable payload written
* Multiple hosts targeted
* After-hours activity
* High data volume spike
* Correlated credential dumping
* SMB write followed by remote execution

---

## 9. Indicators of False Positive

* IT admin performing patching
* Backup solution running
* SCCM deployment
* Software update push
* Approved file migration

---

## 10. Incident Response Actions (If True Positive)

### 10.1 Containment

* Isolate source host
* Disable compromised account
* Block SMB temporarily between segments
* Remove host from network

---

### 10.2 Eradication

* Reset credentials
* Remove malicious files
* Patch exploited systems
* Hunt for persistence mechanisms

---

### 10.3 Recovery

* Restore from backups
* Rebuild infected hosts
* Validate domain integrity
* Monitor SMB traffic post-incident

---

## 11. Mitigation & Prevention

* Restrict SMB east-west
* Disable SMBv1
* Implement SMB signing
* Enforce least privilege
* Monitor administrative share access
* Deploy EDR with lateral movement detection
* Implement network segmentation
* Use conditional access policies

---

## 12. Actions an IR Should Never Do (In Context of Alert)

* Do not immediately block SMB globally (business disruption risk)
* Do not delete suspicious files before forensic collection
* Do not reset accounts without capturing authentication logs
* Do not assume internal SMB is benign
* Do not close alert without validating share path and user context
