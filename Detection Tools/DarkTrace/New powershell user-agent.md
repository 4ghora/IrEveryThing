## 1. Alert Name

**New PowerShell User-Agent**

---

## 2. Alert Description (Detection Logic Explanation)

This alert indicates that a host on the network has initiated outbound network communication using a **PowerShell-specific or PowerShell-like HTTP User-Agent string** that is new or uncommon for that device or the environment.

---

### 2.1 What triggered the alert (Detection Logic)

This alert is typically triggered by **behavioral anomaly detection at the packet, flow, and application-layer (L7) level**.

**Common triggers include:**

* **HTTP/HTTPS requests where the User-Agent header contains PowerShell identifiers**, such as:

  * `Mozilla/5.0 (Windows NT; Windows PowerShell)`
  * `PowerShell/7.x`
  * `Microsoft PowerShell`
  * Empty or malformed User-Agent typical of scripted PowerShell requests
* **Deviation from historical baselines**, where:

  * The host has never used PowerShell for outbound HTTP before
  * The destination domain/IP has never been contacted via PowerShell
* **PowerShell web cmdlets** detected indirectly via traffic patterns:

  * `Invoke-WebRequest`
  * `Invoke-RestMethod`
  * `System.Net.WebClient`
* **Encrypted HTTPS traffic with PowerShell fingerprints**, such as:

  * Short-lived HTTPS sessions
  * Low packet entropy variation
  * Regular beacon intervals

**Detection layers involved:**

* Packet inspection (HTTP headers)
* Flow analytics (session timing, byte ratios)
* Behavioral profiling (per-host application usage)

---

### 2.2 Which security tool generated it

Generated by **Darktrace** using unsupervised machine-learning models that baseline normal application behavior per device.

---

### 2.3 Why this alert is important (Network Security Perspective)

PowerShell is a **dual-use administrative tool** but is heavily abused by attackers for:

* **Command and Control (C2)** communication
* **Payload retrieval (download cradle attacks)**
* **Living-off-the-Land (LOLBins)** techniques that evade traditional signature-based controls

From a network perspective:

* PowerShell traffic blends into **legitimate HTTPS**
* Often bypasses proxy, IDS, and firewall rules
* Used **post-compromise**, indicating a higher likelihood of real intrusion

This alert frequently appears **after initial access**, making it critical for early-stage containment.

---

### 2.4 Severity Definition & Escalation Conditions

**Default Severity:** Medium

**Escalate to High/Critical if any of the following are observed:**

* Communication with:

  * Known malicious IPs/domains
  * Newly registered or low-reputation domains
* Periodic beaconing patterns (C2 behavior)
* PowerShell traffic from:

  * Servers (DCs, DB servers)
  * Privileged user workstations
* Concurrent alerts:

  * Suspicious DNS
  * SMB/WMI anomalies
  * Credential access or lateral movement signals
* Evidence of data transfer (exfiltration-sized payloads)

## 3. Knowledge Required Before Investigation

### 3.1 Network & Protocol Concepts Analyst Must Understand

#### a) HTTP/HTTPS User-Agent Behavior

* The **User-Agent** is an HTTP header identifying the client application.
* Browsers send rich, standardized User-Agents.
* PowerShell-generated requests often:

  * Use minimal or static User-Agents
  * Repeat identical headers across sessions
* Example:

  * Browser: `Mozilla/5.0 (Windows NT 10.0; Win64; x64)`
  * PowerShell: `Mozilla/5.0 (Windows NT; Windows PowerShell)`

#### b) PowerShell Networking Internals

PowerShell uses:

* `.NET Framework` networking classes
* `System.Net.Http.HttpClient`
* `System.Net.WebClient`

These generate **distinct traffic characteristics**, such as:

* Short TCP sessions
* Predictable TLS behavior
* Script-driven request intervals

#### c) Command-and-Control (C2) Over HTTP/S

* Attackers prefer HTTP/S because:

  * Allowed outbound by default
  * Encrypted (limits payload inspection)
* PowerShell is ideal for:

  * Beaconing
  * Receiving encoded commands
  * Downloading second-stage malware

#### d) Living-Off-the-Land (LOLBins)

* PowerShell is a **signed Microsoft binary**
* Often not blocked by application controls
* Network traffic is the **primary detection surface**

## 4. Attacker Perspective

### 4.1 Why attackers use PowerShell User-Agents

* Native to Windows (no dropper needed)
* Bypasses application whitelisting
* Easy to automate and obfuscate
* Blends into legitimate admin activity

---

### 4.2 What attackers try to achieve

* Establish **stealthy C2 channels**
* Download additional payloads
* Execute commands remotely
* Perform reconnaissance and lateral movement
* Exfiltrate data over HTTPS

---

### 4.3 Real-World Attack Examples

* **PowerShell download cradle attacks**
* **Cobalt Strike PowerShell stagers**
* **Empire / Nishang frameworks**
* **Initial payload fetch after phishing**
* **Cloud-hosted C2 (Azure, AWS, GCP)**

---

### 4.4 Potential Business Impact

* Full endpoint compromise
* Credential theft and privilege escalation
* Lateral movement to critical servers
* Data exfiltration
* Regulatory and reputational damage

## 5. Pre-Investigation Checklist

### 5.1 Validate Network Entities

* Source IP → Hostname → User
* Destination IP/domain → Reputation → ASN
* Ports (80/443/custom)
* MAC address and VLAN

---

### 5.2 Network Context Validation

* Network zone:

  * User LAN
  * Server segment
  * DMZ
* Traffic direction:

  * Internal → External (most suspicious)
  * Internal → Internal (possible lateral movement)

---

### 5.3 Alert Correlation

* Check:

  * Prior Darktrace alerts on same host
  * DNS anomalies
  * SMB/WMI events
  * EDR detections

---

## 6. Investigation Steps

### 6.1 Questions Analyst Should Ask

* Is PowerShell usage expected on this host?
* Is the destination legitimate and known?
* Is traffic interactive or beaconing?
* Is this user or system context?
* Are payload sizes consistent with C2?

---

### 6.2 Answers to Investigation Questions

* Admin workstation? Possibly benign.
* Finance/HR endpoint? Highly suspicious.
* Cloud IP with low reputation? High risk.
* Repeating intervals? Strong C2 indicator.

---

### 6.3 Major Investigations

* Inspect User-Agent strings
* TLS session analysis (JA3/behavioral)
* Beacon interval analysis
* DNS query context
* Destination infrastructure profiling
* Endpoint process correlation (PowerShell.exe)

---

### 6.4 Minor Investigations

* Time-of-day analysis
* Payload size consistency
* Geo-location anomalies
* Comparison with peer hosts

---

## 7. Evidence to Collect

* PCAP or session metadata
* Full HTTP headers (if available)
* TLS handshake details
* DNS logs
* Endpoint PowerShell logs
* EDR process tree

---

## 8. Indicators of True Positive

* PowerShell traffic from non-admin user
* Repeated HTTPS beacons
* Communication with unknown domains
* Encoded payload transfers
* Correlated endpoint alerts

---

## 9. Indicators of False Positive

* Known automation scripts
* Patch management systems
* Internal API calls
* Admin maintenance tasks
* Documented PowerShell tooling

---

## 10. Incident Response Actions (If True Positive)

### 10.1 Containment

* Isolate endpoint
* Block destination IP/domain
* Disable compromised account

---

### 10.2 Eradication

* Remove malicious scripts
* Reset credentials
* Remove persistence mechanisms
* Full endpoint scan

---

### 10.3 Recovery

* Restore from clean state
* Validate no lateral movement
* Monitor for re-beaconing

---

## 11. Mitigation & Prevention

* Restrict PowerShell outbound access
* Enforce PowerShell Constrained Language Mode
* Enable deep TLS inspection where feasible
* Monitor PowerShell logging (Script Block, AMSI)
* Baseline admin PowerShell usage

---

## 12. Actions an IR Should Never Do

* Ignore PowerShell traffic as “normal”
* Close alert without endpoint validation
* Block PowerShell globally without scoping
* Rely only on reputation-based decisions
* Assume HTTPS equals benign traffic
