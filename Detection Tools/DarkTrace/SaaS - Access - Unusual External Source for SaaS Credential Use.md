## 1. Alert Name

**SaaS - Access - Unusual External Source for SaaS Credential Use**

---

## 2. Alert Description (Detection Logic Explanation)

### 2.1 What triggered the alert

This alert is triggered by **behavioral anomaly detection** within network telemetry.

At the **flow and behavioral level**, the detection logic typically includes:

* A user account accessing a SaaS platform from:

  * A **new geolocation**
  * A **new ASN (Autonomous System Number)**
  * A **rare IP range**
  * A **VPN/proxy infrastructure**
* Successful authentication following:

  * Multiple failed login attempts
  * Authentication attempts from different geographies in short time (impossible travel)
* Access from:

  * Hosting providers (DigitalOcean, OVH, AWS EC2)
  * Known anonymization networks
  * Residential proxy networks

**Detection type:**

* Behavioral deviation (user baseline deviation)
* Rare external IP analysis
* Geo-anomaly detection
* ASN anomaly detection

Example (Flow level):

* Internal host 10.10.20.15 → login.microsoftonline.com (HTTPS 443)
* External source IP observed by SaaS provider logs: 185.x.x.x (Russia)
* Previously user only logged in from India corporate ISP.

This is not signature-based — it is **entity behavior analytics (UEBA-style network telemetry)**.

---

### 2.2 Which security tool generated it

Generated by:
**Darktrace**

Darktrace uses:

* Self-learning AI models
* Network metadata analysis
* Behavioral baselining of users and devices
* SaaS activity monitoring via API integrations

It correlates:

* Network flows
* SaaS API logs
* Identity telemetry
* External IP reputation & ASN rarity

---

### 2.3 Why this alert is important from a network security perspective

From a network perspective, this may indicate:

* **Credential compromise**
* Initial access via stolen credentials (MITRE T1078 – Valid Accounts)
* Account takeover
* Use of residential proxy infrastructure
* Compromised VPN credentials
* OAuth token abuse

If successful login occurs from rare infrastructure:

* Attacker may begin:

  * Data exfiltration (T1041)
  * Internal reconnaissance
  * OAuth application abuse
  * Email forwarding rule creation
  * SharePoint/Drive data access

SaaS compromise bypasses traditional perimeter firewalls entirely.

---

### 2.4 Define Severity? Conditions to escalate alert to more severity

**Medium Severity:**

* New geo-login
* Rare ASN
* No suspicious activity after login

**High Severity:**

* Login from high-risk country
* Login from hosting provider IP
* Multiple failed attempts followed by success
* MFA bypassed or disabled
* Concurrent impossible travel pattern

**Critical Severity:**

* Data download spikes
* New OAuth app created
* Email forwarding rules added
* Login to privileged/admin account
* Access to critical SaaS (finance, HR, IAM)

Escalate if:

* Access involves C-suite or privileged user
* Access targets regulated environments (PCI/CDE, HR systems)
* Exfiltration behavior detected

## 3. Knowledge Required Before Investigation

### 3.1 Concepts Analyst Must Understand (Detailed)

This alert involves **identity, network telemetry, and SaaS authentication flows**.

An analyst must understand:

---

### A. HTTPS & TLS Handshake (Port 443)

SaaS authentication occurs over HTTPS.

Basic flow:

1. Client → SYN to SaaS IP (TCP 443)
2. TLS handshake (ClientHello, ServerHello)
3. Certificate validation
4. Encrypted HTTP POST /login request
5. Response with session token

NDR tools do not inspect credentials (encrypted).
They analyze:

* SNI (Server Name Indication)
* JA3 TLS fingerprint
* Flow metadata
* IP/ASN reputation
* Traffic volume & timing

Example:
Unusual JA3 fingerprint for corporate user may indicate scripted login via attacker tool.

---

### B. SaaS Authentication Flow (OAuth / SAML / OIDC)

Common SaaS authentication models:

* SAML federation
* OAuth 2.0
* OpenID Connect
* Azure AD login flows

Login may involve:

* login.microsoftonline.com
* accounts.google.com
* Okta endpoints

An attacker using stolen credentials:

* Performs web-based login
* Or uses API-based login
* Or uses automated scripts

If token issued successfully:

* Session cookie created
* Persistent refresh token granted
* Access to SaaS data begins

---

### C. ASN & IP Reputation

An Autonomous System (ASN) represents ISP or hosting provider.

Corporate users usually:

* Use corporate ISP
* Use company VPN

Attackers often:

* Use VPS providers
* Use residential proxy networks
* Use botnets

ASN rarity detection = login from network never seen before for that user.

---

### D. Impossible Travel Concept

If:

* User logs in from India
* 20 minutes later logs in from Europe

Impossible travel detection triggers.

Requires:

* Timestamp correlation
* Geo-IP mapping
* Speed calculation

---

### E. Network Directionality

This is typically:
Internal → SaaS (normal login)
BUT alert focuses on:
External source using valid credentials

Meaning:
Login may have occurred:

* Outside corporate network
* Without VPN
* From external internet

Important to check:

* Was login via corporate VPN?
* Was login direct to SaaS?

---

### F. MITRE ATT&CK Relevance

Relevant techniques:

* T1078 – Valid Accounts
* T1110 – Brute Force
* T1556 – Modify Authentication Process
* T1566 – Phishing
* T1041 – Exfiltration Over C2 Channel
* T1098 – Account Manipulation

## 4. Attacker Perspective

### 4.1 Why attackers use this network technique

Using valid credentials:

* Avoids malware deployment
* Avoids EDR detection
* Bypasses perimeter firewall
* Appears legitimate

Credential-based access is low-noise.

---

### 4.2 What they try to achieve

* Initial access persistence
* Access mailbox for BEC
* Download sensitive files
* Create forwarding rules
* Register malicious OAuth apps
* Move laterally via identity federation

---

### 4.3 Real-world attack examples

* Business Email Compromise (BEC) campaigns
* OAuth app abuse in Microsoft 365
* Token replay attacks
* Phishing leading to SaaS takeover

Examples:

* Compromised Microsoft 365 accounts used to steal invoices
* Google Workspace accounts abused for data theft

---

### 4.4 Potential Business Impact

* Financial fraud
* Data breach
* Regulatory fines
* Intellectual property theft
* Internal privilege escalation

## 5. Pre-Investigation Checklist

### 5.1 Confirm Source/Destination Details

* External Source IP
* ASN
* Country
* Destination SaaS domain
* Username involved
* Time of login
* Was login successful?

Map:

* IP → Hostname (if internal)
* User → AD identity
* DHCP logs
* VPN logs

---

### 5.2 Check Network Segment & Directionality

* Was access via corporate VPN?
* Was it direct internet login?
* Is user remote worker?
* Is IP known corporate egress IP?

---

### 5.3 Verify Alert Severity & Correlation

Check:

* Multiple alerts from same user?
* Other login failures?
* Data exfil alerts?
* Unusual SaaS API activity?
* EDR alerts on user's device?

---

## 6. Investigation Steps

### 6.1 Questions Analyst Should Ask

1. Has this user logged in from this IP/ASN before?
2. Is this IP associated with VPN or hosting provider?
3. Was MFA used successfully?
4. Were there failed login attempts prior?
5. What actions occurred post-login?
6. Is this login correlated with impossible travel?
7. Is there abnormal data transfer post-authentication?

---

### 6.2 Answers to Questions

1. If no historical login → suspicious.
2. Hosting ASN → high risk.
3. MFA bypass → escalate.
4. Multiple failures → brute force attempt.
5. Post-login file download → exfil.
6. Geo mismatch → account compromise.
7. Spike in download → data theft.

---

### 6.3 Major Investigations

* Pull SaaS audit logs
* Review authentication logs
* Check:

  * Token issuance
  * MFA logs
  * Device ID
  * Browser fingerprint
* Check:

  * File download volume
  * Mailbox rules
  * OAuth app registrations
* Investigate ASN reputation
* Correlate with proxy/firewall logs

---

### 6.4 Minor Investigations

* Check user travel status
* Confirm remote work pattern
* Verify ISP change
* Validate legitimate VPN usage
* Compare TLS fingerprint consistency

---

## 7. Evidence to Collect

* SaaS authentication logs
* IP & ASN details
* Geo-location data
* VPN logs
* Firewall logs
* Proxy logs
* MFA logs
* Audit trail of post-login actions
* Packet capture (if available)
* Darktrace model deviation score

---

## 8. Indicators of True Positive

* Login from hosting provider ASN
* Impossible travel detected
* Failed attempts before success
* MFA fatigue pattern
* New device ID
* Data download spike
* Email forwarding rule created
* OAuth app registered

---

## 9. Indicators of False Positive

* User traveling internationally
* Legitimate ISP change
* Corporate VPN IP rotation
* Mobile carrier IP variation
* Recently relocated employee
* Known contractor access pattern

---

## 10. Incident Response Actions (If True Positive)

### 10.1 Containment

* Immediately disable account
* Revoke active sessions
* Revoke OAuth tokens
* Force password reset
* Enforce MFA re-registration

---

### 10.2 Eradication

* Remove malicious mailbox rules
* Remove unauthorized OAuth apps
* Review admin role assignments
* Scan endpoint for malware
* Review phishing emails

---

### 10.3 Recovery

* Restore compromised files
* Notify affected stakeholders
* Monitor account for 30+ days
* Enable conditional access policies

---

## 11. Mitigation & Prevention

* Enforce MFA (preferably phishing-resistant)
* Implement conditional access (geo restrictions)
* Block risky ASNs
* Deploy impossible travel detection
* Restrict legacy authentication
* Monitor OAuth app consent
* Enable CASB monitoring
* Implement identity-based NDR integration

---

## 12. Actions an IR Should Never Do (In Context of Alert)

* Do NOT assume login is malicious without verifying travel
* Do NOT reset password without collecting logs
* Do NOT inform user before containment (may alert attacker)
* Do NOT ignore minor ASN anomalies
* Do NOT close alert without checking post-login activity
* Do NOT rely solely on IP reputation
