## Alert Name

**Device/Attack and Recon Tools**

---

## Alert Description (Detection Logic Explanation)

This alert indicates that a device inside the network has exhibited network behavior consistent with **attack frameworks, penetration testing tools, or reconnaissance utilities**. These may include port scanners (e.g., Nmap), brute-force tools, exploitation frameworks, C2 frameworks, or enumeration scripts.

---

### 2.1 What Triggered the Alert

In **Darktrace**, this alert is typically triggered via **behavioral anomaly detection combined with pattern recognition**, not just static signatures.

Detection may occur at:

* **Packet level**

  * TCP SYN bursts to multiple ports (port scanning behavior).
  * Abnormal TCP flag combinations (e.g., NULL scans, XMAS scans).
  * High rate of ICMP Echo requests across subnet.

* **Flow level**

  * Single host contacting multiple internal IPs over short duration.
  * Repeated failed SMB/RDP/SSH authentication attempts.
  * Beacon-like outbound traffic to rare external IP.

* **Payload level**

  * HTTP User-Agent strings associated with tools (e.g., `sqlmap`, `curl/7.x`, `python-requests`).
  * TLS JA3 hashes matching known C2 frameworks.
  * SMB negotiation anomalies.

Examples of triggering logic:

* Internal host attempts connections to 300 ports across 50 hosts in 2 minutes.
* Device initiates SMB sessions to multiple servers without prior baseline behavior.
* HTTPS connection to rare ASN with consistent 60-second intervals (beaconing).

---

### 2.2 Detection Tool

Generated by:
**Darktrace** (AI-driven NDR platform)

Darktrace uses:

* Unsupervised machine learning
* Peer group behavioral baselining
* Device role modeling
* Protocol anomaly detection

---

### 2.3 Why This Alert Is Important

Recon and attack tools represent:

* Early stage intrusion activity (MITRE TA0043 – Reconnaissance)
* Internal compromise pivoting (TA0008 – Lateral Movement)
* Pre-exploitation staging
* C2 framework deployment
* Data staging/exfil preparation

From a network perspective:

* Recon activity often precedes ransomware.
* Lateral scanning inside flat networks can lead to domain compromise.
* Outbound C2 establishes attacker persistence.

---

### 2.4 Severity & Escalation Conditions

**Base Severity:** Medium–High
(Depends on internal vs external targeting)

Escalate to **Critical** if:

* Tool activity originates from critical subnet (Domain Controllers, PCI/CDE).
* Successful authentication observed after brute-force attempts.
* Large outbound data transfer follows scanning.
* Communication to known malicious ASN/C2 IP.
* Repeated beaconing patterns observed.
* East-west lateral movement detected.

# 3. Knowledge Required Before Investigation

## 3.1 Network & Protocol Concepts Analyst Must Understand

This section is critical.

### A. OSI Model Relevance

Recon tools operate across multiple OSI layers:

* **Layer 3 (Network)** – IP scanning, subnet sweeps.
* **Layer 4 (Transport)** – TCP/UDP port scanning, SYN scans.
* **Layer 7 (Application)** – SMB enumeration, HTTP directory brute-force, LDAP queries.

Understanding which layer the anomaly exists in helps determine:

* Is this connectivity-level recon?
* Is it service enumeration?
* Is it credential abuse?

---

### B. East-West vs North-South Traffic

* **North-South:** Internal → External (internet-bound)
* **East-West:** Internal → Internal

Recon tools used internally indicate:

* Compromised host pivoting.
* Insider misuse.
* Red team activity.

Example:

* Host A in User VLAN scanning Domain Controller subnet → high risk.
* Host scanning internet IPs → possible staging or compromised device.

---

### C. TCP Handshake & Scan Types

Analyst must understand:

| Scan Type    | Behavior                 | Detection Indicator              |
| ------------ | ------------------------ | -------------------------------- |
| SYN Scan     | SYN → SYN/ACK → RST      | High SYN without full handshake  |
| Full Connect | Complete 3-way handshake | Multiple short-lived connections |
| NULL Scan    | No TCP flags             | Suspicious malformed packets     |
| XMAS Scan    | FIN, PSH, URG flags      | IDS signature match              |

Wireshark filters:

```
tcp.flags.syn == 1 and tcp.flags.ack == 0
```

---

### D. SMB / RDP / SSH Enumeration

Common recon protocols:

* **SMB (445)** – Share enumeration, DC targeting.
* **LDAP (389/636)** – AD enumeration.
* **RDP (3389)** – Lateral movement.
* **SSH (22)** – Linux pivoting.

Indicators:

* Multiple SMB session setups without file transfers.
* NTLM authentication failures across many hosts.
* LDAP queries for domain object enumeration.

---

### E. DNS Recon & DGA Concepts

Recon tools may:

* Query multiple subdomains rapidly.
* Generate algorithmic domains (DGA).
* Query internal zone records.

Important logs:

* Zeek dns.log
* Passive DNS

Look for:

* High NXDOMAIN responses.
* Rare domains.

---

### F. TLS & JA3 Fingerprinting

Tools often use:

* Default TLS libraries.
* Unique JA3 hashes.
* Self-signed certificates.

Check:

* JA3 consistency across sessions.
* Rare SNI values.
* Certificate issuer anomalies.

---

### G. Beaconing Behavior

C2 frameworks like:

* Cobalt Strike
* Sliver
* Metasploit

Network traits:

* Fixed interval callbacks (e.g., 60s ± jitter).
* Low data volume per session.
* Long duration TLS connections.

---

### H. Baseline Behavior

Darktrace relies on:

* Device role profiling (Printer vs Workstation vs Server).
* Peer grouping.

A printer performing port scans → anomaly.
A vulnerability scanner appliance performing scans → expected.

# 5. Pre-Investigation Checklist

### 5.1 Confirm Asset Details

* Source IP
* Destination IP(s)
* Ports used
* MAC address
* DHCP logs
* AD user mapping
* Hostname & device type

---

### 5.2 Check Network Segment

* Is source in:

  * Server VLAN?
  * User VLAN?
  * Guest Wi-Fi?
  * PCI zone?
* Traffic direction?

  * East-west?
  * North-south?

---

### 5.3 Correlate Alerts

* Any EDR alerts on same host?
* IDS signatures triggered?
* Firewall deny logs?
* Authentication logs?

---

# 6. Investigation Steps

## 6.1 Questions Analyst Must Ask

* Was TCP handshake completed?
* How many unique hosts were contacted?
* How many ports per host?
* Were authentication attempts made?
* Were connections successful?
* Was data transferred?
* Is destination ASN reputable?
* Does behavior match vulnerability scanner schedule?
* Is there beaconing periodicity?

---

## 6.2 How to Answer

Using:

**NetFlow/IPFIX**

* Count unique destination IPs.
* Calculate byte counts.
* Session duration.

**Wireshark**

```
tcp.analysis.retransmission
frame.time_delta_displayed > 1
```

**Zeek**

* conn.log → handshake status
* smb.log → share enumeration
* ssl.log → JA3 hash

**SIEM Query Example**

```
| summarize dcount(DestinationIP), dcount(DestinationPort) by SourceIP, bin(TimeGenerated, 5m)
```

---

## 6.3 Major Investigations

### A. Deep Packet Inspection

* Check HTTP headers for tool signatures.
* Examine SMB negotiation.
* Inspect TLS certificates.

### B. Flow Analysis

* Duration
* Bytes sent vs received
* Failed vs successful connections

### C. TLS Analysis

* Check SNI
* JA3 fingerprint
* Certificate chain validation

### D. Authentication Logs

* Multiple failed logins?
* NTLM vs Kerberos?

---

## 6.4 Minor Investigations

* DNS logs before connection.
* Proxy logs.
* Firewall policy hit.
* Threat intelligence lookup.
* Reverse DNS of destination IP.

# 8. Indicators of True Positive

* High-volume SYN scans.
* Lateral movement patterns.
* Consistent beaconing intervals.
* Rare external ASN communication.
* Known malicious JA3.
* SMB enumeration across many hosts.
* Data staging before outbound transfer.
* Multiple failed authentications followed by success.

---

# 9. Indicators of False Positive

* Approved vulnerability scanner (e.g., Nessus).
* IT admin running legitimate audit tool.
* Patch management server.
* Internal monitoring appliance.
* Security testing window.
* CDN routing causing IP variation.

---

# 10. Incident Response Actions (If True Positive)

## 10.1 Containment

* Isolate host via NAC.
* Block IP via firewall.
* Disable user account.
* Shutdown switch port.
* Sinkhole malicious domain.

---

## 10.2 Eradication

* Full EDR scan.
* Credential reset.
* Remove persistence.
* Patch exploited vulnerabilities.
* Remove unauthorized tools.

---

## 10.3 Recovery

* Restore from clean backup.
* Monitor for re-infection.
* Increased logging for affected subnet.
* Validate domain integrity.

---

# 11. Mitigation & Prevention

* Network segmentation.
* Strict east-west firewalling.
* Egress filtering.
* Deploy internal IDS.
* Zero Trust Network Access.
* TLS inspection (where compliant).
* Harden SMB & disable NTLMv1.
* Implement least privilege.

---

# 12. Actions an IR Should Never Do

* Do NOT ping suspected C2.
* Do NOT manually browse suspicious IP.
* Do NOT immediately block without capturing evidence.
* Do NOT assume it is red team without validation.
* Do NOT reboot system before memory capture.
* Do NOT alert attacker via abrupt firewall rule change without monitoring.
* Do NOT ignore east-west scans in flat network.
